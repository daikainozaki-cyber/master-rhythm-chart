## PADDAW ビジョン（2026-03-01 壁打ちで確定）

**PADDAWの最終形はオーディオ録音も含む完全なDAW。** 今作っている各アプリ（Master Rhythm Chart, 64 Pad Explorer, 五度圏）はそれぞれPADDAWのモジュール。独立して動くが、最終的に統合される。**モジュールとして作ること自体がPADDAWの設計。**

### 設計思想
- Abletonと違い、**アレンジメントビュー（マスターリズム譜）でもパッドが使える**
- 五度圏は「学習の足場」、パッドが「到達点」（習熟後は五度圏不要に）
- Session View = Sections、Arrangement View = Form → Score View
- 各機能はパネル（モジュール）としてオン・オフ可能

### パネル構成
| パネル | 役割 |
|--------|------|
| Grid | セクション編集（Session View的） |
| Score | マスターリズム譜 SVG表示（Arrangement View的） |
| Pad | ボイシング表示・パッド入力・再生連動点灯 |
| Circle | 五度圏・ダイアトニック連動・調号→パッド点灯 |
| Analysis | Ⅱ-V検出・セカンダリードミナント・バークリー式破線 |

### 入力方法
- キーボード → コード入力・カーソル移動
- PUSH パッド → ボイシング打鍵→コード自動判定→配置
- PUSH ジョグ/エンコーダー → 譜面ナビゲーション
- ピアノロール（Score View上クリック） → キメ打ち込み
- リアルタイム録音 → キメのタイミング記録

### 書き出し
- SVG / PNG — 譜面画像
- .als — Ableton Liveプロジェクト
- .clvz — Clover Chord System（インポートも）
- .chs — 64 Pad Explorer

## コンセプト

**Clover Chord Systemが独占している「マスターリズム譜作成」市場に、Web技術 + パッドボイシング表示で参入する。**

Clover Chord Systemは競合なし・進歩なし。Mac専用（Windows版もあるがWeb版なし）。MIDI再生なし。コード入れ替えが面倒。この隙間に、64パッドアプリで培った技術で入る。

## 仕様（うりなみさんの構想から）

### 第1段階：リズム譜作成（最小MVP）

**コード入力 → D&D配置 → リズム譜表示**

1. **コード入力**: 64パッドアプリのコード入力システムを流用
2. **D&D配置**: コードを小節にドラッグ&ドロップ
3. **グリッド表示**: DAW的なグリッドでリズム譜を表示
4. **コード入れ替え**: ドラッグで移動するだけ（Cloverの弱点を解決）

### 核心機能：パッド入力 → コード譜自動生成

**演奏がそのまま入力になる。Cloverの3ステップ入力を完全に超える。**

```
【Clover】マウスで3クリック（ルート→クオリティ→テンション）→ 配置
【本アプリ】パッドでボイシングを弾く → コード自動認識 → リズム譜に直接流し込み
```

### 爆速入力：カーソル + パッド

**マウスに触らない。片手カーソル、片手パッド。**

```
← →  グリッド位置を移動（スナップ）
↑ ↓  音価を変更（全→2分→4分→8分→16分→3連）
パッド  弾いた瞬間にコード確定・配置
```

操作例：
```
→ → ↓ ↓ [Cm7弾く] → → [F7弾く] → ↓ [Bb7弾く]
 = Cm7(8分) . F7(4分) . Bb7(8分) ...
```

D&Dは後から並べ替える時に使う。入力自体はカーソル+パッドが最速。

64 Pad Explorerのコード判定機能がそのまま使える：
- パッドを押さえる → ボイシングからコード名を自動判定
- 判定されたコードが「入力バッファ」に入る
- 再生中にリアルタイム入力 or 後からD&Dで配置
- **弾いたボイシング（手形）がそのままコードに紐づく** → 手形表示の精度が100%

### 小節内の操作

```
【デフォルト】
コードを小節に投入 → 拍1に四分音符として配置

【発音位置の調整】
拍内にドラッグ → グリッドにスナップ
  - 偶数系グリッド（8分、16分）
  - 3連系グリッド

【音価の調整】
右端をドラッグ → 音符の長さを伸縮
  - アンティシペーション = 音を伸ばして次の小節にかかる

【入力なし】
リズム入力がなければデフォルト四分音符
```

### 第2段階：再生 + パッドボイシング表示

- リズム譜に沿って進行が自動再生
- パッド上にボイシング（手形）がリアルタイム表示
- テンポ調整

### 第3段階：判定機能

- 64パッドアプリの判定機能を統合
- リズム譜の進行に同期して「正しいボイシングを押せてるか」判定

### Cloverファイルインポート（Phase 3.5で実装済み）

- .clvzファイル（gzip圧縮XML）を読み込んで本アプリ形式に変換
- `fileio.js` の `importClvz()` で実装

## 本質：3つの用途が一つのアプリに統合される

```
シーケンサ    → コードを時間軸に並べて再生する
リズム譜      → コード進行を譜面として表示する
練習ツール    → 進行に合わせて手形を表示して判定する
```

同じデータ、同じUI、視点が違うだけ。

**= PAD DAWの準備。** 簡易シーケンサーを入れることで、PAD DAWへの段階的な道ができる。

### 三面等価とパッドの位置づけ（2026-03-02）

```
五度圏 = ハーモニーの視点（何のコードか）→ 教育的入口
64 Pad = ボイシングの視点（どう鳴らすか）→ 中心（パッドは五度圏を物理配置として内包）
リズム譜 = 時間の視点（いつ鳴らすか）→ 応用
```

音楽を三面から見ているが、教育的には等価ではない。**パッドが中心**。パッドのレイアウト自体が五度圏であり、理論が「見える」楽器。五度圏はパッドを知らない人のための補助ビュー。

**「3アプリのタブ統合」ではなく「1つの音楽体験の3つのビュー」。** 共有される結節点（コード状態、オーディオエンジン、トランスポート）を通じて接続する。

### UI設計原則：操作体系の一貫性

**「入口は複数、出口は一つ」** — コードビルダー、インクリメンタル入力、メモリー、ダイアトニックは全て**コードビルダーの状態をハブとして経由**し、同一のコードオブジェクトとしてグリッドに出る。

AIの弱点: 機能単位で個別に作ってしまい、論理的結節点（ハブ）が見えない。分類（ツリー）は得意だが結合のトポロジー（グラフ）が見えない。

**詳細**: `memory/ui-design-principles.md`

**用途：**
1. **うりなみさんの制作ツール** — LOD、Song of the Month等のコード進行を作る日常ツール。自分で使うから改善が止まらない
2. **HPS教材配信** — 作ったリズム譜をそのまま会員に共有
3. **会員の練習ツール** — 手形を見ながらパッドで練習、判定付き

## ビジネスモデル：ツール無料 / コンテンツHPS専用

**うりなみさんの決定（2026-02-09）：ツールは無料でしか出さない。**

```
【無料（誰でも）】ツール本体
  - リズム譜作成・編集
  - コード入力（パッド/手入力）
  - D&D配置・カーソル爆速入力
  - 再生・パッドボイシング表示
  - .clvzインポート

【HPS専用（有料メンバーシップ）】コンテンツ
  - ストックボイシング（よく使われるかっこいいボイシング集）
  - LOD / Song of the Month のリズム譜データ
  - レッスン教材
```

**構造の強み：**
- ツール = 器（無料で配る → 認知度・信用度）
- コンテンツ = 中身（HPS専用 → 収益）
- ツールは全機能使える。機能制限なし → ユーザーに嫌われない
- ストックボイシング = うりなみさんの「理論×演奏×経験」の結晶 → コピー不可能
- ツールがコピーされても、コンテンツのキュレーション（「この進行でこの押さえ方がかっこいい」）は真似できない

**技術実装：**
```
ツール本体 → GitHub Pages（認証なし）
HPS専用データ → HPSポータル経由（Cloudflare Access認証）
```
同じアプリ、データ層だけで切り分け。HPS会員がログインするとストックボイシングが追加で見える。

**全ツール共通モデル（2026-02-09 確定）：**
- 64 Pad Explorer → 無料（全機能）/ ストックボイシング・トップノート分析はHPS専用
- リズム譜アプリ → 無料（全機能）/ ボイシングデータ・教材はHPS専用
- 将来のツール → 同じ構造で展開

## スケール/アプローチ提案機能（2026-02-18 壁打ち）

### 背景: 64 Pad Explorerに入れられなかったもの

64 Pad ExplorerのAvailable Scaleはコード・スケール理論（縦: 1コード→対応スケール）ベース。ペンタトニック/ブルーススケールはモード（横: キー全体に敷く）であり、この仕組みに乗らない。

しかし実戦ではペンタが最初に手が伸びるもの。ソウル・ファンクでは中途半端にドリアン等を使うとらしさを失う。**コード・スケール理論だけではカバーできない音楽の世界**がある。

### リズム譜アプリでの実装構想

リズム譜アプリには横の流れ（コード進行の時間軸）がある。ここなら：

- **進行全体を見て**「この区間はペンタで通せる」「ここはトライアド+クロマチック」と提案できる
- コード・スケール理論の提案（64 Pad Explorer的な縦の選択肢）も併記できる
- **ジャンル/スタイルに応じた切替**: ジャズ→コード・スケール理論中心、ファンク/ソウル→ペンタ/ブルース/ソウルスケール中心

### 提案するアプローチの種類

| アプローチ | 適用場面 | 備考 |
|-----------|---------|------|
| メジャーペンタ | ほぼ全コード | トーナリティベース |
| マイナーペンタ | ほぼ全コード | トーナリティベース |
| ブルーススケール | ブルース進行、ファンク | b5追加 |
| ソウルスケール | ソウル/R&B | |
| トライアド+クロマチック | 全般 | |
| コード・スケール（縦） | ジャズ | 64 Pad Explorerの機能を継承 |

### うりなみさんの本領が活きる場所

ファンク・ソウルの実践知——これはうりなみさんの参入障壁の核心。コード・スケール理論のツールは他にもあるが、**ペンタ/ブルース/ソウルスケールの実践的な使い分けをツール化したものは存在しない**。

## 反復・ナビゲーション記号の意味と再生ルール（テストで学習中）

**僕（蔵人）の弱点**: 記号から再生順序を逆算するのが苦手。うりなみさんのテスト問題で一つずつ覚えていく。

### リピート記号

| 記号 | 名前 | 意味 |
|------|------|------|
| `\|\|:` | リピート開始 | ここから繰り返し区間が始まる |
| `:\|\|` | リピート終了 | ここで繰り返し区間が終わる。開始位置に戻って2回目を演奏 |

**基本ルール**: リピート終了に到達 → リピート開始に戻る → 2回目を通過したらそのまま先へ進む

**パターン①**: A | ||: B | C :|| | D → **A-B-C-B-C-D**（B-Cを2回通る）

**パターン②**: A | B :|| ||: C | D :|| → **A-B-A-B-C-D-C-D**（2つの独立したリピートブロック）
- :|| の前に ||: がない → **曲頭に戻る**
- :|| ||: が隣接 → 2つのリピートブロックの境界

### エンディング（1番カッコ / 2番カッコ）

| 記号 | 意味 |
|------|------|
| `1.` | 1回目のみ演奏する小節 |
| `2.` | 2回目のみ演奏する小節（1番カッコを飛ばしてここへ） |

### ナビゲーション記号

| 記号 | 名前 | 意味 |
|------|------|------|
| 𝄋 | セーニョ (Segno) | ジャンプ先の目印。曲中に1つ |
| D.S. | ダル・セーニョ | この小節を演奏し終えたらセーニョに戻る |
| D.C. | ダ・カーポ | この小節を演奏し終えたら曲頭に戻る |
| Fine | フィーネ | D.S./D.C.で戻った2回目のパスでここに到達したら演奏終了 |
| 𝄌 | コーダ (Coda) | コーダセクション開始の目印 |
| to𝄌 | トゥ・コーダ | D.S./D.C.パス中にここを通過したらコーダにジャンプ |

**再生エンジンの状態**: `playDSActive` フラグでD.S./D.C.後のパスを判別。Fine/toCodaはこのフラグがtrueの時のみ有効。

**パターン⑥**: A-B-C-D(𝄋)-E-F-G(Fine)-H(D.S.) → **A-B-C-D-E-F-G-H-D-E-F-G**
- 1回目: A→Hまで通常再生、HのD.S.でセーニョ(D)に戻る
- 2回目: D→G、GのFineで停止（playDSActive=trueなので有効）

**パターン①（エンディング版）**: A | ||: B | [1. C :||] | [2. D] → **A-B-C-B-D**
- 1回目: B→C(1番カッコ)→:||でBに戻る
- 2回目: B→C(1番カッコ)をスキップ→D(2番カッコ)→終了
- **各カッコは必ず1回しか通らない**（うりなみさんの教え）

**パターン②（3番カッコ）**: A ||: B [1. C D :||] [2. E F :||] [3. G H] → **A-B-C-D-B-E-F-B-G-H**
- 1回目: B→C-D(1番)→:||でBに戻る
- 2回目: B→E-F(2番)→:||でBに戻る
- 3回目: B→G-H(3番)→終了
- `playPass++` でリピートのたびにパスが進む。最大エンディング番号までリピート

**D.C.（ダ・カーポ）**: A-B-C-D-E(Fine)-F-G-H(D.C.) → **A-B-C-D-E-F-G-H-A-B-C-D-E**
- D.C.で**曲頭**に戻る。Fineで停止

**D.S.（ダル・セーニョ）**: A-B-C(𝄋)-D(Fine)-E-F-G-H(D.S.) → **A-B-C-D-E-F-G-H-C-D**
- D.S.で**セーニョの位置**に戻る。Fineで停止

**D.C. vs D.S.の違い**: 戻り先だけ。D.C.=曲頭、D.S.=セーニョ。Fineの動作は同じ

**D.S. al Coda（コーダ付き）**: A-B-C(𝄋)-D(to𝄌)-E-F-G-H(D.S.)  𝄌I-J-K
→ **A-B-C-D-E-F-G-H-C-D-I-J-K**
- 1回目: A→Hを通常再生（toCodaはplayDSActive=falseなので無視して通過）
- D.S.でSegno(C)に戻る → playDSActive=true
- D.S.パス: C→D（toCodaが発動 → Dを演奏後にCoda(I)へジャンプ）
- コーダ: I→J→K → 終了
- **toCodaは「その小節を演奏してからジャンプ」**（演奏前ではない）
- **toCodaの位置 = D.S.パスで最後に演奏する小節**
